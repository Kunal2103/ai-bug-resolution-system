# 🚀 AI Bug Resolution System (Assessment 2)

An autonomous, multi-agent AI system built with **LangGraph** that ingests bug reports, navigates codebases, analyzes logs, and programmatically reproduces and patches software failures.

* **Input Mode Declared:** Option A (Provided Mini-Repo)

## 📖 Overview
Engineering teams frequently receive bug reports with incomplete context. This system acts as an automated Senior Engineering task force. It uderstands the report, investigates the environment dependencies, hunts down the relevant source code, searches the logs, writes and executes a minimal reproducible test script, and generates a structured patch plan. 

## 🧠 Multi-Agent Architecture
This system utilizes a deterministic **LangGraph** state machine to orchestrate 7 specialized AI agents. Data is passed via a shared `GraphState`, ensuring explicit and safe handoffs.

1. **Triage Agent:** Extracts symptoms, expected vs. actual behavior, and prioritizes initial hypotheses.
2. **Dependency Analyst:** Inspects configuration files (`requirements.txt`, `pyproject.toml`) to rule out environment/versioning conflicts before analyzing code.
3. **Repo Navigator:** Autonomously explores the directory structure using sequential tool-calling to pinpoint the exact source files containing the bug.
4. **Log Analyst:** Uses context from the Navigator to execute targeted regex searches over system logs, extracting stack traces and anomalies.
5. **Reproduction Agent:** Acts as an automated QA engineer. It writes a minimal reproduction script (`repro_test.py`), executes it in the environment, and loops to self-correct its own code until the target bug is successfully triggered.
6. **Fix Planner:** Proposes a root-cause hypothesis, patch approach, and validation plan based on the verified reproduction.
7. **Reviewer Agent:** Challenges assumptions, ensures the fix is safe, and formats the final output into a structured YAML report with a calculated confidence metric.

## 🏆 Addressing Evaluation Criteria 

* **Engineering Judgment:** Implemented prompt-level constraints to prevent LLM "parallel tool-calling" hallucinations. The Repo Navigator is forced to work sequentially (read directory -> wait -> read file), ensuring verified paths over guessed paths.
* **Sound System Design:** A strictly **Config-Driven Design**. No hardcoded file paths, models, or parameters. Everything is dynamically loaded from `config/config.yaml`.
* **Reproducibility:** The system doesn't just guess the fix; it physically proves it. The `Reproduction_Agent` programmatically writes and runs `repro_test.py` to guarantee the bug exists in reality, not just in theory.
* **Attention to Safety & Failure Modes:** - **API Throttler:** Built-in exponential backoff and a configurable `api_delay` to gracefully handle HTTP 429 Rate Limit errors without crashing the pipeline.
  * **Infinite Loop Protection:** The self-correcting execution engine is capped by a `max_iterations` limit to prevent runaway token consumption.
  * **Token Defenses:** Agents are explicitly instructed to ignore massive auto-generated files (like `.lock` files) during environment analysis.
* **Clear Evaluation:** The system utilizes *Functional Evaluation*. It judges its own success by reading the actual terminal `stdout/stderr` generated by its test scripts, rather than relying solely on LLM text generation. 

## ⚙️ Setup & Installation

**1. Clone the repository**
```bash
git clone <your-repo-url>
cd AI_Bug_Resolution_System
```

**2. Set up the environment**

This project uses uv for **lightning-fast dependency** management.
```uv venv
# This will automatically create the .venv and install exact dependencies from uv.lock
uv sync

# activate the environment
source .venv/bin/activate  # On Windows: .venv\Scripts\activate
```

**3. Configure API Keys**
Create a .env file in the root directory and add your Groq API key:
```bash
GROQ_API_KEY=your_api_key_here
```

## ▶️ Running the System
To launch the multi-agent workflow, simply execute:
```bash
uv run python main.py
```

**1. Inputs Location**   

  The system automatically pulls the buggy mini-repo, the markdown bug report, and the system logs from the **data/inputs/** directory.

**2. Outputs Location**
   
  Upon successful execution, the system generates two critical artifacts in the **data/outputs/** directory:

   - **repro_test.py**: The runnable minimal reproduction script generated by the AI.

   - **final_resolution.yaml**: The structured report containing the bug summary, log evidence, root cause hypothesis, and patch plan.
     
     ## 🐛 Verifying the Minimal Repro
   
     - The system generates a minimal reproducible test.
     - You can run this standalone script to verify the bug exists.
     
     #### Command to run:-
     
         uv run python data/outputs/repro_test.py
     
     #### Expected Failing Output:-
     
     - The script is designed to fail consistently and should produce the exact error caught by the **Log Analyst**:
     
            Traceback (most recent call last):
            
            File "data/outputs/repro_test.py", line ...
       
            ...
       
            IndexError: list index out of range

##  🔍 Traceability and Logging
To ensure complete transparency of agent decisions, this system uses a **custom centralized logger**. When running main.py, the **console will output a clear trace of:**

   - Which agent is currently active.

   - Which specific tools are being called (e.g., Tool 'list_directories' called to scan repository path: '.').

   - The exact files identified by the Repo Navigator.

   - API throttling pauses and retry attempts.

This centralized logging ensures full **traceability**, providing a transparent audit trail of the AI's decision-making process.
